## MODIFIED Requirements

### Requirement: Gemma-backed image description
The system SHALL generate the final `describe_image` text using the multimodal LLM checkpoint `google/gemma-3-27b-it` by providing the image as input alongside an instruction prompt.

The returned description MUST:
- be a single dense paragraph,
- use a dry, descriptive tone (no flowery language),
- cover (when visible) subject, pose, clothing, lighting, background, and camera angle,
- avoid inventing details; if a detail is unclear, it MUST be described as unclear/indistinct.

The system MUST treat the `max_length` parameter as the output token budget for generation (`max_new_tokens`) and MUST default `max_length` to 300 output tokens when not provided.

The system MUST require a CUDA-capable device for this operation; if CUDA is not available, the operation MUST fail rather than attempting CPU execution.

#### Scenario: Successful description
- **WHEN** a task requests `type=describe_image`
- **THEN** the system returns a natural-language description generated by Gemma

#### Scenario: Default max_length
- **WHEN** a task requests `type=describe_image` without a `max_length` param
- **THEN** the system generates using an output token budget of 300 (`max_new_tokens=300`)

#### Scenario: No GPU available
- **WHEN** a task requests `type=describe_image` and CUDA is not available
- **THEN** the system returns an error result for that task

## REMOVED Requirements

### Requirement: Non-fatal fallback
**Reason**: `describe_image` no longer uses BLIP/base captions, and returning a weaker fallback caption is undesirable for training-dataset quality.
**Migration**: Clients MUST handle `describe_image` task failures (status="error") and/or ensure the service runs with a CUDA-capable GPU for Gemma multimodal inference.
